# 识别任务停止与终止功能设计

## 1. 背景与目标

当前 Web UI 将整理任务放入后台线程执行，OCR 提供者在后台线程内使用异步并发识别。任务开始后，用户只能等待识别完成或失败，无法主动中断。

本功能增加两个操作：

- **停止识别**：不再启动新的文件识别，等待当前正在识别的文件结束。
- **终止任务**：立即取消本地尚未完成的识别任务，不再等待远端 OCR 结果。

两种操作均保留已完成文件的识别结果并允许预览，但只要本次任务存在未完成文件，就禁止导出，避免产生不完整的报销包。

## 2. 范围

### 2.1 包含

- Web UI 单报销包模式。
- Web UI 批量报销包模式。
- 任务状态查询与页面轮询。
- OCR 并发任务的协作式取消。
- 已完成结果保留及未完成文件状态展示。
- 停止、终止和并发边界测试。

### 2.2 不包含

- 不为 CLI 增加交互式停止命令；CLI 仍使用 `Ctrl+C`。
- 不强制终止 Python 后台线程。
- 不保证取消 PaddleOCR 服务端已经创建的 Job；终止后仅停止本地等待和结果接收。
- 不支持从停止点继续原任务。用户需要重新提交任务。
- 不允许部分导出。

## 3. 用户交互

### 3.1 按钮显示

任务处于 `queued`、`running` 或 `stopping` 状态时，在进度区域显示：

- `停止识别`
- `终止任务`

任务进入最终状态后隐藏或禁用两个按钮。

### 3.2 停止识别

用户点击 `停止识别` 后：

1. 前端向任务停止接口提交请求。
2. 后端将取消模式设为 `stop`。
3. 页面状态显示“正在停止识别，等待当前文件完成”。
4. 系统不再启动尚未开始的文件识别。
5. 当前正在识别的文件允许自然完成。
6. 未启动文件标记为 `已停止`。
7. 所有进行中的文件结束后，任务进入 `stopped` 状态。

停止请求不需要二次确认，因为该操作不会中断正在执行的识别。

### 3.3 终止任务

用户点击 `终止任务` 后：

1. 页面弹出二次确认，明确说明未完成文件不会继续等待且本次任务不能导出。
2. 用户确认后，前端向任务终止接口提交请求。
3. 后端将取消模式设为 `terminate`。
4. 系统取消本地尚未完成的异步识别任务，并停止等待远端 OCR Job。
5. 等待中或识别中的文件标记为 `已终止`。
6. 任务进入 `terminated` 状态。

如果远端服务不支持 Job 取消，已经提交的 Job 可能仍在服务端运行，但本地不再轮询或接收其结果。

### 3.4 最终结果

`stopped` 和 `terminated` 状态下：

- 已完成文件保留识别类别、金额和风险信息。
- 页面允许查看已完成结果的预览。
- `确认导出` 按钮保持禁用。
- 页面恢复 `开始整理` 按钮，允许创建新任务。
- 页面明确显示任务未完整完成及不可导出的原因。

## 4. 后端设计

### 4.1 取消控制对象

新增轻量级、线程安全的取消控制对象，供 Web 任务、流水线和 OCR 提供者共享。对象至少提供：

- 当前取消模式：`none`、`stop`、`terminate`。
- 请求停止的方法。
- 请求终止的方法；`terminate` 优先级高于 `stop`，停止后仍允许升级为终止。
- 查询是否禁止启动新工作的接口。
- 查询是否应立即取消工作的接口。

控制对象只负责传递取消意图，不直接结束线程。

### 4.2 任务状态

任务新增以下状态：

- `stopping`：已收到停止请求，等待当前识别完成。
- `stopped`：停止完成。
- `terminating`：已收到终止请求，正在取消本地任务。
- `terminated`：终止完成。

任务内部保存取消控制对象，但任务快照必须继续过滤以下划线开头的内部字段，不能将线程同步对象序列化到前端。

### 4.3 HTTP 接口

新增两个任务操作接口：

- `POST /tasks/{task_id}/stop`
- `POST /tasks/{task_id}/terminate`

接口规则：

- 任务不存在时返回明确的未找到错误。
- 仅对 `queued`、`running`、`stopping` 状态接受请求。
- 重复提交相同请求时保持幂等。
- `stop` 后允许提交 `terminate`，将温和停止升级为立即终止。
- 任务已经完成、失败或进入最终取消状态时，不改变结果，并返回当前任务快照。

### 4.4 流水线取消边界

`organize_folder`、`organize_batch_subfolders` 和内部解析流程接收可选取消控制对象。

单报销包模式：

- 启动每个文件识别前检查取消状态。
- `stop` 时不启动新文件，等待已启动文件完成。
- `terminate` 时取消尚未完成的本地 Future 或异步 Task。
- 只对成功完成的文件生成记录。
- 取消后仍基于已完成记录构建预览，但不进入正常 `review` 状态。

批量报销包模式：

- 启动下一个报销包前检查取消状态。
- `stop` 时不再启动新的报销包，当前报销包按温和停止规则收尾。
- `terminate` 时立即终止当前报销包的本地等待，并不再处理后续报销包。
- 尚未处理的报销包显示为 `已停止` 或 `已终止`，不可导出。
- 即使某个报销包在取消前已经完整识别，本次批量任务仍禁止整体或逐包导出，避免形成同一次任务中的部分交付。

### 4.5 OCR 取消

`SdkOcrProvider.parse_many` 接收可选取消控制对象。

- 创建异步任务前检查是否允许启动。
- 等待提交限速、重试退避和远端轮询期间定期响应终止请求。
- 收到 `stop` 后，不创建新的 `process_one` 工作；已进入处理的任务继续。
- 收到 `terminate` 后，对未完成的 `asyncio.Task` 调用取消，并使用 `gather(..., return_exceptions=True)` 完成清理。
- `asyncio.CancelledError` 作为控制流处理，不转换成普通 OCR 失败记录。
- 已经完成并成功转换的结果正常返回。

由于当前 SDK 的单次 `parse_document` 可能内部包含提交和轮询，实际终止响应时间取决于 SDK 对协程取消和 HTTP 请求取消的支持。目标是尽快停止本地等待，不承诺服务端 Job 同步消失。

## 5. 数据与状态展示

文件状态新增：

- `已停止`
- `已终止`

任务完成数量只统计已得到最终识别结果的文件，不将 `已停止` 和 `已终止` 计入成功完成数量。界面仍显示文件总数，使用户能够看出本次任务不完整。

预览数据只包含已完成识别并生成记录的文件。页面同时显示未完成文件数量，避免用户误认为预览覆盖了全部票据。

## 6. 竞争条件与错误处理

- 如果停止请求到达时某个文件刚好完成，以实际已生成的记录为准，该文件保留为完成状态。
- 如果终止请求与 OCR 完成同时发生，只保留终止请求前已经被流水线接收的结果；取消后到达的结果不再写入任务。
- 如果任务在请求到达前已自然完成，则保持 `review` 或 `done`，不回退为停止状态。
- 停止过程中可以升级为终止；终止不能降级为停止。
- 取消操作本身发生异常时，任务进入 `failed`，错误信息说明取消失败，不伪装成正常停止。
- 不删除已经写入的原始结果文件；最终任务不可导出，且页面明确标识结果不完整。

## 7. 测试与验收标准

### 7.1 后端测试

- 停止请求使未启动文件不再进入 OCR。
- 停止请求等待正在执行的文件自然结束。
- 终止请求取消未完成异步任务，并在有限时间内结束本地任务。
- 已完成记录在停止和终止后继续存在。
- 未完成文件分别显示 `已停止`、`已终止`。
- 停止后可以升级为终止。
- 重复停止或终止请求幂等。
- 已完成任务收到取消请求时状态不变。
- 单报销包和批量报销包均覆盖。
- `stopped`、`terminated` 状态均不可导出。

### 7.2 前端测试

- 运行时显示两个按钮。
- 点击停止后按钮状态和提示正确。
- 点击终止前出现二次确认。
- 最终取消状态会停止轮询并恢复 `开始整理`。
- 最终取消状态保留已完成文件和预览。
- 最终取消状态禁用导出。

### 7.3 灰度验证

使用真实 Web UI 入口和可控的慢速 OCR 替身分别验证：

1. 多文件任务启动后点击停止，确认当前文件结束、后续文件不再启动。
2. 多文件任务启动后点击终止，确认页面快速进入终止状态。
3. 检查已完成记录仍可预览。
4. 检查未完成文件状态准确且导出按钮不可用。
5. 在批量模式中确认后续报销包不再启动。

## 8. 成功标准

- 用户可明确选择温和停止或立即终止。
- 后台线程不被强制杀死，任务状态和输出文件保持一致。
- 取消操作最终可收敛到明确状态，不长期停留在运行中。
- 已完成结果可见，未完成范围可辨识。
- 任何不完整任务均无法导出。
